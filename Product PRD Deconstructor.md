# Product PRD Deconstructor

## 适用场景

当用户出现以下需求时立即调用本技能：

- “给你一个链接，帮我拆解这个产品”
- “分析这个产品的付费功能和技术结构”
- “做一个详细PRD给我”
- “补充这个产品的背景、由来、竞品、发展脉络”
- “输出能直接复制到飞书的文档”

## 输入要求

最少输入：

- 产品链接（官网、产品页、应用商店页、介绍页之一）

可选输入：

- 目标受众（如：老板汇报、投研、竞品分析、产品方案）
- 输出深度（轻量/标准/深度）
- 是否需要对比竞品

如用户未给全信息，默认使用“深度分析 + 可落地PRD”模式直接执行。

## 执行流程

1. 读取用户提供的产品链接内容，提取核心功能、定位、人群、定价线索。
2. 补充互联网背景信息：公司背景、创始故事、融资/里程碑、市场背景、主要竞品。
3. 拆解功能体系，尤其识别并标注付费功能：
   - 免费与付费边界
   - 订阅层级与权益差异
   - 付费转化路径（触发点、限制点、升级引导）
4. 推断技术结构：
   - 客户端形态（Web/iOS/Android/桌面/插件）
   - 后端能力（鉴权、内容服务、计费、推荐、通知、日志）
   - AI/数据能力（模型调用、RAG、向量检索、特征工程、埋点分析）
   - 基础设施（云服务、CDN、对象存储、消息队列、监控告警）
5. 生成“可直接复制进飞书”的PRD正文，结构固定为：
   - 公司背景
   - 商业层
   - 技术层

## 输出格式约束（必须遵守）

- 一级标题必须带 emoji
- 内容必须按以下三大模块输出，不得打乱：
  - `# 🏢 公司背景`
  - `# 💰 商业层`
  - `# 🛠️ 技术层`
- 文风：专业、可汇报、可落地，避免空话
- 表格：每个模块至少一个高价值表格（优先三线逻辑的结构化表格）
- 输出应可直接复制到飞书，无需再二次整理

## 结果质量要求

- 必须区分“已验证事实”与“合理推断”
- 对不确定信息标注“待验证”
- 不编造具体营收/融资数值；若无可靠来源，给区间或定性判断
- 重点强调付费功能设计逻辑与商业闭环，不只列功能名

## 调用后行为

- 若用户仅给链接：直接完成完整拆解并输出PRD
- 若用户给多个链接：先做统一抽象，再做分产品对比表
- 若用户要求“更细”：补充用户旅程、功能流程、指标体系（AARRR、漏斗、留存）
